# CodeNova 개발 원칙

## 핵심: 극단적 미니멀리즘

모든 코드에 "왜 존재하는가?"를 묻는다. 정당화되지 않으면 삭제한다.

## 개발 철학

- 처음에 쓸만한 수준을 만들어서 디벨롭한다. 완벽한 설계보다 동작하는 결과물을 우선한다

## 코드

- 외부 npm 패키지보다 Node 내장 모듈을 우선한다 (`fs.readdir` > `fast-glob`, `child_process` > `simple-git`)
- `dependencies`는 0에 가까울수록 좋다. 패키지 추가 전 내장 API로 해결 가능한지 반드시 확인한다
- interface, type alias, 상수 추출을 하지 않는다. 인라인과 TS 추론에 위임한다
- 한 번만 쓰이는 변수는 인라인한다. 중간 변수 대신 체이닝한다
- "나중에 필요할 수도" 있는 코드는 작성하지 않는다
- import는 named import 선호 (`{ readdir, readFile }` > `fs`). 대체 가능한 import는 제거한다 (`path` → 템플릿 리터럴)

## 변경

- 요청과 직접 관련된 코드만 수정한다. 인접 코드, 주석, 포맷을 "개선"하지 않는다
- 기존 스타일을 따른다. 다르게 하고 싶어도 맞춘다
- 자기 변경으로 생긴 고아(미사용 import/변수/함수)만 정리한다. 기존 dead code는 언급만 한다

## 실행

- 구현 전에 가정을 명시한다. 불확실하면 묻는다
- 해석이 여러 개면 골라서 진행하지 말고 제시한다
- 작업을 검증 가능한 목표로 변환한다 ("버그 수정" → "재현 테스트 작성 → 통과시키기")
- 여러 단계 작업은 단계별로 검증 기준을 명시한다

## 커밋

- Co-Authored-By 금지
- 의존성이 변경되면 기존 커밋을 리셋하고 깨끗하게 다시 커밋한다
- 커밋 메시지는 영어로 짧고 명확하게 작성하며, 태그를 포함한다 (`feat:`, `fix:`, `refactor:`, `chore:`, `test:`, `docs:` 등). 태그 뒤에 스코프를 붙이지 않는다 (`feat(core):` ✗ → `feat:` ✓)
- 작업 내용을 세분화하여 논리적 단위로 나눠서 커밋한다. 하나의 커밋에 여러 관심사를 섞지 않는다
- PR 제목과 본문은 영어로 작성한다
- PR 본문은 `.github/pull_request_template.md` 템플릿(What / Why / Test)을 사용한다
- push 전에 `act`로 CI를 로컬에서 실행하고 통과를 확인한다. 실패 시 수정 후 재실행한다

## 커뮤니케이션

- 한국어 사용
- 코드를 제안하기 전에 불필요한 것부터 제거할 수 없는지 먼저 검토한다
- "이건 필수입니다"라고 말할 때는 왜 대체 불가능한지 근거를 제시한다
